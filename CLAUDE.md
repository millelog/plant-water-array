## Project Overview

Plant Water Array — distributed plant moisture monitoring system with ESP32 sensors, a Python backend, and a React frontend. ESP32 devices self-register via captive portal WiFi provisioning and send periodic moisture readings. Code updates are pushed to devices over the LAN via `deploy.py`.

## Development Commands

### Backend (FastAPI + SQLite)
```bash
cd backend
source .venv/bin/activate          # venv lives at backend/.venv
pip install -r requirements.txt
uvicorn main:app --reload --host 0.0.0.0 --port 8000
```
API docs at http://localhost:8000/docs

### Frontend (React + Vite + TypeScript + Tailwind)
```bash
cd frontend
pnpm install
pnpm dev          # Vite dev server on port 5173
pnpm build        # TypeScript compile + Vite bundle
pnpm lint         # ESLint
```

### ESP32 Flashing (initial setup only)
```bash
python flash_esp.py               # Auto-detect port, erase, flash MicroPython, upload source
python flash_esp.py --sync-only   # Upload source files without re-flashing firmware
python -m mpremote connect <port> repl   # Serial console debug
```
Windows: use `python -m esptool` (not `esptool.py` directly — version mismatch issue).

### Deploying Code Updates to Devices
```bash
python deploy.py                  # Push embeded-src/*.py to all registered devices over LAN
python deploy.py --dry-run        # Show what would be deployed without pushing
python deploy.py --force          # Deploy even with uncommitted changes
python deploy.py --server URL     # Override backend URL (default: DEPLOY_SERVER env or http://192.168.70.222:8000)
python deploy.py --port 8266      # Override device update server port (default: 8266)
```
Flow: reads `.py` files from `embeded-src/` (excluding `config.py`), gets device list + deploy tokens from backend API, pushes files to each device's update server (port 8266), then sends apply command with git commit hash. Devices reboot after applying.

## Architecture

Three independent components, no monorepo tooling:

**Backend** (`backend/`) — FastAPI, SQLAlchemy ORM, SQLite (`backend.db`, auto-created).
- Entry: `main.py` → mounts routers from `routers/` (devices, sensors, readings, alerts, zones, dashboard, config, watering_logs, admin)
- `models.py` — ORM models: Device, Sensor, Reading, Threshold, Alert, SystemConfig, Zone, WateringLog, HeartbeatLog
- `schemas.py` — Pydantic request/response models
- `crud.py` — all DB operations
- `database.py` — SQLite connection setup + idempotent migrations

**Frontend** (`frontend/`) — React 18, TypeScript, Tailwind CSS, Radix UI, Recharts.
- Routes in `src/routes.tsx`: Dashboard, Compare, Devices, DeviceDetail, Sensors, Readings, PlantDetail, Alerts, Settings
- API client in `src/api/api.ts` — all backend calls via axios, base URL from `VITE_API_URL` env or `http://localhost:8000`
- Pages in `src/pages/`, reusable components in `src/components/`
- Path alias: `@` → `src/`

**Embedded** (`embeded-src/`) — MicroPython on ESP32. Note the directory is intentionally spelled `embeded-src`.
- `main.py` — WiFi connect, device/sensor registration, reading loop, heartbeat, update server
- `update_server.py` — Non-blocking HTTP server (port 8266) for receiving push deploys
- `captive_portal.py` — AP mode provisioning (SSID: `PlantSensor_XXXX`, pw: `plantsetup`)
- `config.py` — device-specific config, gitignored, generated by captive portal
- `_version.py` — auto-generated on deploy, contains `GIT_COMMIT` hash (not in repo)

**Deploy** (`deploy.py`) — CLI script at project root. Pushes code to devices over LAN.

## Data Flow

1. ESP32 boots → connects WiFi (or starts captive portal for provisioning)
2. Starts update server on port 8266 for receiving push deploys
3. Registers itself via `POST /devices/register_device` (sends deploy_token), then registers sensors via `POST /sensors/`
4. Main loop: reads moisture → `POST /readings`, heartbeat every 300s, checks for incoming deploys (non-blocking)
5. Backend checks thresholds on new readings → creates alerts if exceeded
6. Frontend polls backend APIs to display devices, readings, alerts
7. Developer runs `python deploy.py` → pushes code to all online devices, devices reboot with new code

## Deploy System

- **Push model**: developer runs `deploy.py` which pushes files to devices (replaces old pull-based OTA)
- **Auth**: each device gets a unique 32-char hex `DEPLOY_TOKEN` during captive portal provisioning
- Tokens are stored in device `config.py` and sent to backend during registration
- `deploy.py` fetches tokens from backend API (`GET /devices/`) — no manual token management
- Devices verify token before accepting any file push
- Files are staged as `.new`, SHA-256 verified, then swapped in atomically with backup/restore on error
- Version tracking: `_version.py` written on apply with git commit hash, reported in heartbeats

## Key Conventions

- Package manager for frontend is **pnpm** (not npm)
- Backend uses **pip** with `requirements.txt` (no pyproject.toml)
- Database is SQLite — schema defined in `models.py`, tables auto-created, new columns added via `database.py:upgrade_db()`
- Device cascade delete: deleting a device removes its sensors, readings, and alerts
- CORS is fully open (`*`) in backend — development only
- `config.py` is protected — never overwritten by deploy or push
